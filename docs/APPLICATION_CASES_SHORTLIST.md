# 应用案例 · 选型与首批清单（调研 + 决策）

> 版本：v1 · 2026-06-19 · 配套 `docs/PRD_APPLICATION_CASES.md`（本文件是 PRD §5「候选案例目录」的**落地决策版**）
> 状态：选型决策，待评审；实现按 PRD 分期进行。本文件 doc-only，基于 `main`（与 v0.8.0 独立）。

---

## 1. 选型标准（用户定的两条硬指标）

每个案例必须**同时**满足下面两条，才进清单：

- **标准一 · 典型 + 学生易懂 + 生活常用**：必须是学生**亲身接触过、一听就懂**的场景——背后能对应一个学生熟悉的产品/触点（手环、小爱同学、垃圾分类桶、手机拦截骚扰短信…）。抽象、离生活远、需要专业背景才懂的，一律不选。
- **标准二 · 与既有通用技术架构一脉相承**：必须落在**现有 6 个 `ai_capability` 之一**，复用同一套**四步流程 + ML 引擎 + 数据导入 + 教育字段**。**零新能力、零新流程**——案例只是给现有技术换一个「真实场景」的外壳。

> **数据就绪度不是选型标准**（用户没提）。它只用来排**分批先后**（见 §4）：典型且一脉相承的案例先定下来，再按「数据现成 / 需采集 / 依赖检测」决定谁先做。

---

## 2. 调研：现有可复用的资产（一脉相承的锚点）

平台已有的 6 个能力，以及各自**已经存在的「应用味」样本包**（`data_packs/`）——这些就是「一脉相承」的现成抓手，也直接决定了哪些案例数据现成：

| 现有能力 | 数据类型 | 已有「应用味」样本包（`data_packs/`） | 数据现成？ |
|---|---|---|---|
| `image_classifier` | image | 仅 `basic_shapes`（抽象）+ 采集指引卡（无内置图） | ✗ 需策展/采集 |
| `text_classifier` | text | **垃圾信息识别**、情绪识别、天气分类、新闻主题 | ✓ 4 个现成 |
| `audio_classifier` | audio | 无（音频无法内联，靠采集助手） | ✗ 学生自采 |
| `qa_retrieval` | qa | **校园生活问答**、科学常识问答 | ✓ 2 个现成 |
| `sensor_decision_model` | sensor | **运动状态**、健康监测、环境舒适度 | ✓ 3 个现成 |
| `object_detector_trainable` | detect | 仅 `basic_shapes`（抽象） | ✗ 需标注 + YOLO |

**结论**：文本 / 问答 / 传感器三类，已有现成且贴近生活的样本包，可零摩擦起步；图像 / 语音 / 检测需要采集或策展。

## 3. 选型决策：每个能力一个「旗舰典型案例」

**原则**：和技术层一一对应（6 能力 → 6 旗舰案例），每个挑**最典型、学生最有感**的那一个。这样案例层完整镜像技术层（一脉相承最彻底），学生「学完图像分类 → 去图像分类的旗舰案例」路径清晰。

| 能力（一脉相承） | 旗舰典型案例 | 为什么典型（对应的生活触点） | 数据来源 | 四步映射（与通用任务**完全一致**） | 备选案例 |
|---|---|---|---|---|---|
| `image_classifier` | **垃圾分类**（可回收/厨余/有害/其他） | 垃圾分类是国策、校园天天宣传、还有「智能垃圾桶」——最具代表性的「AI 进生活」 | 需策展四类垃圾图 | ①拍/导入各类垃圾图 ②训练图像分类 ③拍一件新垃圾测 ④导出可运行包 | 拍照识花/识水果（"拍照识图"） |
| `text_classifier` | **垃圾短信拦截**（正常/骚扰广告） | 人人手机都在拦截骚扰短信（12321、手机管家）——一眼就懂的二分类 | **现成**（`general_text_spam`） | ①导入正常/垃圾短信 ②训练文本分类 ③输入一条新短信测 ④导出 | 网购评论好差评（情感分析） |
| `audio_classifier` | **语音指令**（"开灯/关灯/播放音乐"） | 小爱同学、小度、Siri——喊一声就执行，学生天天用 | 学生自采（采集助手） | ①录各类指令音 ②训练语音分类 ③现场喊一句测 ④导出 | 环境声音（警报/玻璃碎） |
| `qa_retrieval` | **校园问答助手** | 智能客服 / 公众号自动回复——问一句答一句 | **现成**（`general_qa_school`） | ①导入问答对 ②建检索库 ③问一个新问题测 ④导出 | 博物馆讲解问答 |
| `sensor_decision_model` | **运动计步**（静止/走路/跑步） | 手环 / 手机「今日步数」——每个学生都见过 | **现成**（`general_sensor_motion`） | ①导入三轴加速度 CSV ②训练决策模型 ③输入一组新读数测 ④导出 | 健康监测（体温心率→提醒） |
| `object_detector_trainable` | **车牌检测**（定位车牌框） | 停车场 / 小区抬杆——学生坐车都见过 | 学生标注 + 外部数据 | ①画框标注车牌 ②训练检测（**建议 YOLO26**）③传新图看框 ④导出 | 口罩佩戴检测（更简单） |

> 检测案例只做「定位车牌框」，不读车牌号（读号是检测+OCR 流水线，留三期）；且靠 v0.8.0 的 **YOLO26** 才对路、依赖 `.[detect]` + YOLO 验证/导出分支——详见 `PRD_APPLICATION_CASES.md` §6.1。

## 4. 分批落地（按数据就绪度排先后，**不改选型**）

选型已定（§3），只按数据摩擦决定先做谁：

- **首批 · 数据现成、立即可做**（验证案例层走得通）：
  1. **垃圾短信拦截**（text，样本包现成）
  2. **校园问答助手**（qa，样本包现成）
  3. **运动计步**（sensor，样本包现成）
  → 三个一上来就能一键导入、几秒训练、跑通四步，最适合先打通「应用案例」基础设施。

- **次批 · 典型度最高、需采集/策展**：
  4. **垃圾分类**（image，需策展四类垃圾图——最具旗舰意义，优先补数据）
  5. **语音指令**（audio，学生自采，天然走采集助手）

- **后批 · 依赖检测/YOLO**：
  6. **车牌检测**（detect，依赖 YOLO26 实跑验证 + 导出分支 + 车牌数据，见 PRD）

## 5. `case_scenario` 文案种子（实现时直接用）

每个案例首页卡片/详情页要突出的「一句话痛点」：

- 垃圾分类：「让机器看一眼就知道这是哪类垃圾，扔对桶。」
- 垃圾短信拦截：「让手机自动认出骚扰短信，帮你拦下来。」
- 语音指令：「喊一声『开灯』，机器听懂并执行。」
- 校园问答助手：「学生问一句，机器自动找到最接近的答案。」
- 运动计步：「靠传感器读数，判断你是站着、走着还是在跑。」
- 车牌检测：「在画面里自动框出车牌的位置（停车场抬杆第一步）。」

## 6. 没选什么、为什么（守住「典型」这条线）

| 排除项 | 理由 |
|---|---|
| `basic_shapes`（形状分类/检测） | 抽象、非生活场景，是技术练习不是「应用案例」。留作技术层入门，不进案例层。 |
| 天气描述分类 | 偏「按词找类」的文字游戏，不如垃圾短信直观「有用」。 |
| 情绪识别（文本） | 不如「评论好差评」具体（好差评对应真实的网购评价场景）。降为备选。 |
| 新闻主题分类 | 典型（推荐流），但「拦截骚扰短信」对学生更有切身感。降为备选。 |
| 环境舒适度 / 健康监测 | 都典型，但「运动计步」是学生最普遍、最一眼懂的传感器场景。降为备选。 |

原则：**同一能力下，只把最典型、最有切身感的那个升为旗舰**，其余作备选，避免案例层堆成「样本包列表」而稀释「典型」。
