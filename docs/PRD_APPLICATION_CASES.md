# 应用案例（学以致用）PRD / 技术方案 — Lite DeepLearning Studio

> 版本：v2 · 2026-06-18（v1 基于 v0.7.0；本版基于 **v0.8.0** 重做，纳入目标检测算法 B「YOLO26」的影响）
> 状态：**规划中（本文仅规划，未写功能代码）**。作为下一里程碑 **v0.9.0** 的产品/技术方案（v0.8.0 已用于检测算法 B）。
> 关联决策：本期「应用案例」= **复用现有 AI 能力 + 真实场景数据 + 应用情境包装**，复用现有四步流程与 ML 引擎；不新增 `ai_capability`；多能力串联流水线（如车牌「检测+OCR识别字符」全流程）不在本期范围，列为后续旗舰案例。

## 0. 定位与约束

### 0.1 两层学习结构

平台当前**只有一层组织方式：按技术分类**。首页「机器学习任务」区把 6 个通用任务按 `ai_capability` 排列（图像分类 / 文本分类 / 语音分类 / 智能问答 / 传感器决策 / 目标检测），学生学的是「**一种技术怎么训练**」。

本 PRD 在其之上加**第二层：按实用、具体的典型案例组织**，引导学生「**把学过的技术用到一个真实场景里走完整四步**」（学以致用）。

| | 一期 · 技术分类（已有） | 二期 · 应用案例（本 PRD） |
|---|---|---|
| 组织维度 | `ai_capability`（图像分类 / 文本分类…） | 真实应用场景（垃圾分类 / 车牌检测 / 语音指令…） |
| 学生学到 | 「怎么训练一个 X 模型」 | 「把 X 用到一个真实问题里」 |
| 入口 | 首页「机器学习任务」区 | 首页新增「应用案例 · 学以致用」区 |
| 数据 | 自采或导入任意数据 | 尽量自带「一键导入」的整理数据集，贴合场景 |
| 叙事重心 | 技术原理（特征、过拟合…） | 真实痛点、谁在用、做出来能干嘛 |

两层**并存、互链，不替换、不重构一期**。教学弧：**先学会工具，再把工具用到真实场景**。

### 0.2 本期硬边界（与「案例本质」决策一致）

- **复用现有能力，不新增 `ai_capability`**：每个案例的 `ai_capability` 指向现有 7 种能力之一。
- **复用现有四步骨架与 ML 引擎**：案例走现有 `/project/{id}` 四步页（分类走 `app.js`，检测走 `detect.js`），训练/预测走现有 `app/ml/engine.py`。CPU 友好、几秒出结果的课堂体验保持不变。
- **检测案例现在有两种算法可选**（v0.8.0 起）：轻量检测（算法 A，CPU 几秒）与 **YOLO26 端到端（算法 B，需 `.[detect]`）**——见 §6.1，对车牌这类自定义目标尤其重要。
- **本期不做多能力流水线**：完整车牌识别（检测定位 + OCR 识别字符）这类「串联多个模型」的案例需要新「管线」基础设施，列为**三期旗舰案例**，本期不做。
- **本期产出仅为本规划**；批准后再按 §10 分期实现。

> 版本说明：当前产品版本 **v0.8.0**（`pyproject.toml` / `CHANGELOG.md` / `CLAUDE.md` §0）；`app/main.py` 的 `ASSET_VERSION = "0.13.0"` 仅为 CSS/JS 缓存号，非产品版本。应用案例作为 **v0.9.0** 规划。

### 0.3 v0.8.0 带来的、与本 PRD 相关的变化

v0.8.0（提交 `f86a04f`，「目标检测二期·算法 B」）有几处直接影响应用案例：

1. **目标检测算法 B「YOLO26」落地**（`app/ml/detect_yolo.py`）：Ultralytics YOLO26n 端到端微调，端到端学「找框 + 认框」、免 NMS。引擎按 `model_choice` 训练（lite/yolo）、按 `model_meta.algorithm` 预测。**这让「车牌检测」这类自定义目标真正可检测**（不再受限于 SSD 候选框）。
2. **`.[detect]` 可选组件**（`ultralytics>=8.3.0`，~2GB）：YOLO 卡片仅在装了 `.[detect]` 时变可训，否则显示安装提示（同 `.[ocr]` 降级）。未装也能用算法 A 走完四步。
3. **深度学习地图第 3 节新增「检测演进」**（R-CNN → YOLO → YOLO26/RF-DETR，`templates/learn_deep_learning.html`）：可作为案例「技术 → 应用」的现成教育桥。
4. **Windows 自包含分发包**（`packaging/build_windows_bundle.py`，内嵌 Python + 离线 wheels + 一份精简 lite 数据集子集）+ `build_release.py` 纳入 `data_packs/`：为案例自带数据集提供了现成的**分发通道**。
5. **两个重要 caveat（写进风险）**：YOLO26 **尚未在装 `.[detect]`（+显卡）的机器实跑验证**；**检测导出包目前只覆盖算法 A**，YOLO 导出分支待做——故「用 YOLO 训练的检测案例」第 4 步导出尚不完整。

## 1. 为什么要「应用案例」这一层

**纯按技术分类的局限**：学生学完「图像分类怎么训练」，未必知道「这能拿来干嘛」。技术是手段，不是目的。竞赛与真实世界都是**从问题出发**（「怎么自动分垃圾」「怎么认出车牌」），而不是从「我要用图像分类」出发。

**应用案例把技术落到问题上**：
- 给技术一个**真实的去处**，回答「学这个有什么用」。
- 用**贴合场景的真实数据**降低「不知道拿什么数据练」的门槛（一键导入整理好的案例数据集）。
- 为日后**竞赛向整合**铺路——竞赛题本质就是「带评分标准的应用案例」。

**其实平台已有雏形**：`data_packs/` 里的样本包（情绪识别、天气分类、新闻主题、垃圾信息识别、校园问答、科学常识问答…）已经是「应用情境包装」的数据。本 PRD 把这种零散的「应用味」数据，**升级成一等公民的「案例」入口**——有自己的卡片、叙事、一键数据、四步引导。

---

## 2. 「应用案例」是什么（技术本质）

一个案例 = **现有能力 + 整理好的真实数据 + 应用叙事**。它**复用**以下全部现成基建，新增的只是「目录 + 少量字段 + 首页一区 + 案例内容」：

| 复用项 | 现有实现 |
|---|---|
| 四步骨架 | `templates/project.html` + `static/app.js`（检测案例走 `static/detect.js`），四步顺序锁不变 |
| ML 引擎 | `app/ml/engine.py` 按 `ai_capability` 分派 train/predict（检测再按 `model_choice` 分 lite/yolo），**完全复用** |
| 数据一键导入 | `app/services/dataset_library.py` + `ProjectService.import_platform_dataset()`（按 `ai_capability` 匹配、每类抽样、替换语义） |
| 教育字段 | `TaskDefinition` 的 `concept_intro / real_world_examples / common_mistakes / hands_on_experiments / next_steps`（v0.4.0 起，已在 workflow.html / project.html 渲染） |
| 导出包 | `app/services/template_service.py` + `export_service.py`，按能力渲染（检测目前只覆盖算法 A），**复用** |

差别只在**叙事重心**：技术任务讲「特征 / 过拟合 / 算法对比」；案例讲「这个场景为什么适合用 AI、行业里谁在用、做出来能解决什么问题」。

## 3. 数据模型设计（推荐方案，本期不实现）

### 3.1 新增案例目录 `APPLICATION_CASES`

在 `app/task_catalog.py` 中，与 `GENERAL_TASKS` 平行新增一个 `list[TaskDefinition]`，用一个 `CompetitionDefinition` 容器包裹：

```python
APPLICATION_CASES: list[TaskDefinition] = [ ... ]

APPLICATION_CASES_GROUP = CompetitionDefinition(
    slug="application_cases",
    title="应用案例",
    tasks=APPLICATION_CASES,
)
```

**始终可见、不进 edition 过滤**——和 `GENERAL_ML` 同款处理：`get_competition()` 现已对 `GENERAL_ML.slug` 特判直接返回（`app/task_catalog.py:829`，绕过 `list_competitions(edition)` 的竞赛过滤），对 `application_cases` 加同样一行特判即可。`get_task("application_cases", task_slug)` 随之可用。

### 3.2 复用 `TaskDefinition` + 新增 2 个可选字段

每个案例**就是一个 `TaskDefinition`**，`ai_capability` 指向现有能力。建议在 `app/models.py` 给 `TaskDefinition` 加 **2 个可选字段**（默认空，不影响现有任务）：

| 新字段 | 类型 | 用途 |
|---|---|---|
| `case_scenario` | `str = ""` | 这个案例要解决的真实问题/痛点，一句话，首页卡片与 workflow 页突出展示 |
| `bundled_dataset_id` | `str = ""` | 指向 `dataset_library` 里整理数据集的 id；步骤一可一键导入，为空则学生自采 |
| `case_domain`（可选） | `str = ""` | 领域分组用（环保 / 交通安防 / 校园 / 农业 / 医疗健康 / 生活），供 `/cases` 分组与筛选 |

> 这三个字段对现有技术任务无影响（默认空），是**纯增量**。`group` 字段继续做年级/类型标签（如「通用练习」），不复用、不改语义。
> 检测案例可选定默认算法：用 `case_scenario` / `step_guides` 文案引导（如车牌案例建议 YOLO26），引擎已支持 `model_choice` 选择，无需新增字段。

### 3.3 路由复用

- 案例详情页复用现有 `GET /workflow/{competition_slug}/{task_slug}` → `competition_slug="application_cases"`。
- 创建项目后复用现有 `GET /project/{project_id}` 四步页。
- 即：**零新路由**，只多一个被特判为「始终可见」的 competition slug。

### 4.1 首页新增「应用案例」区

## 6. 用户点名的两个案例 · 边界澄清（诚实标注）

### 6.1 车牌检测（v0.8.0 后更新）

- **本期范围**：目标检测案例，**只做「检测/定位车牌框」**（在图上框出「车牌在哪」），不读车牌号。完整「定位 → 裁剪 → OCR 识别字符」是多能力串联流水线，列为 **三期旗舰案例**，本期不做。
- **算法选择（v0.8.0 关键变化）**：
  - 车牌是**自定义目标**，预训练 SSD 很可能提不出车牌候选框 → **算法 A 轻量检测对车牌效果差**（这正是 `PRD_OBJECT_DETECTION.md` 风险表里「候选框召回」的局限）。
  - **算法 B「YOLO26」端到端自己学找框**，才是车牌定位的对路做法。所以车牌案例**建议默认 YOLO26**（`model_choice="yolo"`），学生标注若干车牌框后微调。
- **三个 caveat（必须在案例文案/风险里讲清）**：
  1. YOLO26 需 `.[detect]`（~2GB，CPU 分钟级、建议显卡）；未装时只能用算法 A（车牌效果有限）。
  2. **检测导出包目前只覆盖算法 A**，YOLO 导出分支待做 → 用 YOLO 训练的车牌案例**第 4 步导出尚不完整**（需等 YOLO 导出分支）。
  3. YOLO26 训练/推理**尚未实跑验证**（见 `CLAUDE.md` §0）→ 车牌案例落地前需先验证 YOLO 路径。
- **数据**：可用现有画框标注工具让学生自标少量车牌；外部车牌检测数据集需检测三期的 COCO 导入器。
- **结论**：车牌案例**技术上现在可行（靠 YOLO26）**，但**就绪依赖三件事**：YOLO 实跑验证、YOLO 导出分支、车牌数据。故建议**晚于首批纯分类案例**上线。

### 6.2 OCR

- **本期**：复用现有 `ocr_typo_checker` 能力做应用案例（在已有「非遗知识卡片查错」之外，可加「试卷/作业查错」「商品标签文字核对」等场景叙事）。
- 在线步骤二仍是**文本比对**（现状），真实图片 OCR 仍在导出 `run.py`（`.[ocr]` / EasyOCR）——**沿用现状，不改 OCR 实现**，案例只做场景包装与引导。

## 7. 数据策展与分发依赖（内容侧）

案例要「学生一上来就能跑通」，关键在**自带整理好的数据集**。这部分**复用现有机制**：

- 数据放在 `LDS_DATASETS_ROOT`（默认 `datasets/`，gitignored，见 `CLAUDE.md` §12）下，按 `docs/DATASET_FORMAT.md` 的 `platform_dataset.json` 清单格式组织，按 `ai_capability` 匹配。
- 案例 `bundled_dataset_id` 指向某个清单 id；步骤一一键调用 `import_platform_dataset()` 导入（已有每类容量上限、随机抽样、替换语义）。
- **分发**：v0.8.0 的 `build_windows_bundle.py` 已内嵌「精简 lite 数据集子集」、`build_release.py` 已纳入 `data_packs/` → 案例自带数据集有了**现成分发通道**，可挂在同一打包流程里随安装包下发。
- **「为首批案例策展数据集」是内容侧依赖**，由老师/策展侧产出（与代码并行）。代码侧只需读取，不写数据集。
- 检测类案例（车牌等）的外部数据额外依赖**检测数据导入器**（COCO → 平台框，`PRD_OBJECT_DETECTION.md` 三期），就绪较晚；可先靠学生自标少量框起步。

## 8. 教育内容整合

案例**复用现有教育字段**，但叙事换挡到「应用」：

| 字段 | 技术任务里讲 | 案例里讲 |
|---|---|---|
| `concept_intro` | 技术原理（特征/过拟合） | 这个**场景**为什么适合用 AI |
| `case_scenario`（新） | — | 要解决的**真实痛点**（一句话） |
| `real_world_examples` | 该技术的应用举例 | 这个场景**行业里谁在用**、长什么样 |
| `common_mistakes` | 通用数据坑 | 这个场景**特有**的坑（如车牌光照/角度） |
| `hands_on_experiments` | 加数据/换特征对比 | 场景化实验（换光照、加难样本…） |
| `next_steps` | 通往深度学习地图 | 通往**更难的案例**或对应深度学习知识 |

**现成的教育桥**：v0.8.0 已在深度学习地图第 3 节加了「检测演进」（R-CNN → YOLO → YOLO26）。检测案例（车牌等）的 `next_steps` 可直接指向那里，把「车牌为什么要用 YOLO」讲透——技术演进与应用选型在此天然衔接。

## 11. 非目标

- 不重构现有「技术分类」任务，不改其组织方式。

## 12. 风险与开放问题

| 问题 | 说明 / 建议 |
|---|---|
| 数据集就绪是瓶颈 | 案例价值高度依赖「自带整理数据集」。首批只上**数据已就绪**的案例（含已有样本包），其余随策展推进。 |
| 与技术任务重复感 | 同一能力会同时出现在「技术」和「案例」两层。靠**叙事 + 自带数据 + 互链**区分，避免学生觉得只是换名字。 |
| 案例数量膨胀 | 靠 `case_domain` 分组页 + 难度筛选（二期）控制首页拥挤。 |
| **车牌案例 = YOLO 依赖链** | 车牌靠算法 B YOLO26 才对路，但 YOLO **未实跑验证**、**导出分支未做**、车牌数据未备。三者任一未就绪都会让车牌案例「能训不能导」或「跑不起来」→ 与首批分类案例**解耦排期**，验证通过再上。 |
| `.[detect]` 安装门槛 | YOLO 需 ~2GB 组件 + 建议显卡，课堂机未必装得起。案例文案要讲清：未装时退回算法 A（车牌效果有限），分类案例完全不受影响。 |
| 车牌「只定位不识别」的预期管理 | 文案讲清本期只框车牌、不读车牌号，避免落差；全流程入三期。 |
| `group` vs 新字段语义 | `group` 继续做年级/类型标签，新增 `case_*` 字段独立，勿混用。 |

## 13. 成功指标（定性）

1. 学生能说出平台的两层结构：「**一期学技术，二期把技术用到真实场景**」。
2. 学生进一个案例，能**一键导入**贴合场景的数据、几秒训练（分类案例）、看到**贴合场景**的结果、导出可运行包。
3. 学生能从一个技术任务**自然走到**对应案例（互链生效），也能从案例点回「了解原理」/深度学习地图。
4. 老师能在不写代码的前提下，照着「应用案例」给学生布置一个**有真实意义**的小项目。

---

## 14. 一句话总结

**在「按技术分类」之上加一层「按应用案例分类」——案例 = 现有能力 + 真实数据 + 应用情境包装，复用现有四步、引擎与数据导入，零新能力、CPU 友好、可立即落地；先上数据已就绪的首批分类案例（一期），车牌检测案例靠 v0.8.0 新增的 YOLO26 才对路、但要等 YOLO 验证与导出分支就绪（与首批解耦排期），多能力流水线（车牌全流程等）留三期。** 让学生从「学会一种技术」走到「把技术用到真实问题」，也为日后竞赛向整合铺好路。
