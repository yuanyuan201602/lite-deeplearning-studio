> 版本：草案 v1 · 2026-06-13
> 状态：**供讨论，未锁定实现**（按约定，先出方案，确认后再动手）

---

## 0. 定位与约束

用户决定：**把目标检测当作一个独立模块来做**，前端对齐现有四步风格，做好后整合进现有前端；过程中**每一步都嵌入教育内容**，让学生理解四步里每一步的「工作 / 原理 / 效果」。

这份方案要先回答一个绕不开的硬约束，再谈怎么做。

### 硬约束：CPU 课堂里怎么「训练」检测？

平台的灵魂是 **CPU 友好、课堂里几秒到几十秒出结果、不需要显卡**（见 `agent.md`：「对象检测，除非某个竞赛任务明确需要，否则推迟完整 YOLO 集成」）。

但「真正的目标检测训练」（YOLO / SSD 微调）：
- 需要 GPU；在 CPU 上微调一个检测模型要**几十分钟到几小时**；
- 依赖重（PyTorch / Ultralytics，~2GB）；
- 一次训练几十个 epoch，完全没有「点一下、几秒出结果」的课堂感。

**直接照搬「真检测训练」会破坏平台体验。** 所以本方案的核心，是设计一条 **CPU 友好、几秒可训、又能讲清检测原理** 的路径。

## 1. 检测和分类的根本区别（为什么不能直接套用四步）

| | 图像分类（已有） | 目标检测（本模块） |
|---|---|---|
| 回答的问题 | 「这**是**什么？」 | 「**有哪些**东西、各自**在哪**？」 |
| 输出 | 一个类别 | 多个框（位置 x/y/w/h）+ 每个框的类别 + 置信度 |
| 标注 | 给整张图贴一个标签 | 在图上**画框**并给每个框贴标签 |
| 评估 | 准确率 | 框准不准（IoU）、漏检 / 误检、置信度 |
| 难点 | 认得准 | 既要找得到、又要框得准、还要认得对 |

→ 四步流程的**骨架可以复用**（准备数据 → 训练 → 测试 → 导出），但**每一步的内部都要重做**：第 1 步要画框标注工具，第 2 步是不同的训练逻辑，第 3 步要在图上画预测框，第 4 步导出检测脚本。这正是「独立模块」的原因。

## 2. 核心技术决策：三条路线

### 方案 A（推荐）· 候选框（预训练）+ 学生训练的「框分类器」

把检测拆成两步，**完美复用平台已有的图像分类基建**：

```
找候选框（预训练，不训练）  →  认每个框是什么（学生训练，几秒）
```

1. **找框**：用平台已集成的预训练 SSD-MobileNet（`object_detector.py` 已有）产出「这里好像有个东西」的候选框（类别无关的 objectness）。
2. **标注**：学生在自己的图上画框、贴标签（新的前端标注工具）。
3. **训练**：平台把每个框**裁剪**出来 → 用 **MobileNet 提特征**（正是图像分类已在用的 `image_classifier` 那套）→ 训练一个 **sklearn 分类器**。外加一类「背景」（用没框中的随机区域当负样本），让模型学会拒绝无关框。**这一步是普通分类，CPU 上几秒完成。**
4. **预测**：预训练 SSD 在新图上提候选框 → 裁剪 → MobileNet 特征 → 学生的分类器给每个框贴标签（背景类直接丢弃）→ NMS 去重 → 画框。
5. **导出**：打包「框分类器（joblib）+ SSD onnx + run.py」，run.py 做 提框→裁剪→分类→NMS。复用 `ai_runtime/core.py` 模式。

**为什么推荐：**
- ✅ 训练＝在裁剪图上做分类，**复用 image_classifier 全套**（MobileNet 特征 + sklearn），**CPU 几秒**，守住课堂体验。
- ✅ 教学叙事极清晰：**「检测 = 先找候选框，再认每个框是什么」**——四步天然对应。
- ✅ 零新增重依赖（SSD onnx + onnxruntime 已在用）。
- ✅ 守住平台不变量：在线训练的模型 ↔ 导出包可一致（都是 MobileNet 特征 + sklearn）。

**诚实的局限（要在教学里讲明，反而是教学点）：**
- ⚠️ 只能检测「SSD 大致能提出候选框的东西」；对 SSD 完全没见过的奇异目标，候选框可能提不出来。
- ⚠️ 精度比不上真 YOLO。**但这恰好是绝佳的教学对照**：「想要更强的检测，就要像 YOLO 那样端到端训练——而那需要显卡」，直接呼应深度学习地图。
- 缓解：候选框阶段除 SSD 外，可叠加一个轻量「滑动窗口 / selective-search-lite」兜底（可选，二期）。

### 方案 B · 真 YOLO 微调（隔离在可选依赖里，作为「进阶/老师」路径）

用 Ultralytics YOLOv8n 之类在 `.[detect]` extra 里微调。**最「真」，但 GPU/慢/重**，不适合课堂主路径。

→ **定位为二期的「进阶模式」**：老师在有显卡的机器上离线训练，或作为导出包里的「想做真检测看这里」的进阶脚本。不进课堂主流程。

### 方案 C · 纯预训练，不训练

把 `/playground/detect` 升级成正式任务：学生只上传图、看 COCO 80 类的识别结果。**太被动**，不符合「训练你自己的模型」的四步精神。

→ 可作为**第 0 步的「先体验」**入口，但不是模块主体。

### 结论

**主推方案 A**（候选框 + 框分类器），把方案 C 当「先体验」、方案 B 当「进阶/导出脚本里的彩蛋」。下面的四步与架构都按方案 A 展开。

## 3. 数据与标注

### 3.1 数据格式（对齐已整理的检测数据集）

外部目录有 6 个检测数据集，均为 **COCO 格式**（已确认）：
```
<dataset>/
  classes.txt                  # 每行一个类别名
  images/train/*.jpg  images/valid/*.jpg
  annotations/train.json       # COCO: images[] + annotations[bbox,category_id,image_id] + categories[]
  annotations/valid.json
```
- `bbox = [x, y, w, h]`（像素坐标）。
- 平台可写一个 COCO → 平台框存储 的导入器，**复用 `dataset_library` 那套「从整理好的数据集导入」**，把这 6 个数据集接进来（数据侧的「暂不支持」就此解锁）。

### 3.2 平台内框存储（新）

```
workspace/projects/<id>/dataset/
  detect_images/<uuid>.jpg            # 上传的原图
  detect_labels.json                  # [{image, boxes:[{x,y,w,h,label}], width, height}]
```

### 3.3 标注工具（本模块最大的新前端件）

- 在第 1 步：上传图片 → 在 `<canvas>` 上**拖拽画框** → 给框选类别（复用类别管理 UI）→ 保存。
- 支持：画框、移动 / 删除框、缩放适配、每类不同颜色、键盘删除。
- 对齐现有 `static/app.js` / `collect.js` 的风格，独立成 `static/detect.js`，做好后整合。

## 4. 四步流程 × 教育内容

每一步都配「了解这一步」（工作 / 原理 / 效果），和现有任务的 `step_guides` 一致风格。

### 第 1 步 · 准备数据（画框标注）
- **工作**：上传图片，在每张图上把目标**框出来**并贴标签；或一键导入整理好的检测数据集。
- **原理**：检测比分类多了「在哪」。框（bounding box）就是「在哪」的答案；框得准不准、贴得全不全，直接决定模型上限。漏标一个目标，模型就学不会它。
- **效果**：得到一批「图 + 框 + 标签」，这是检测训练的原料。

### 第 2 步 · 训练模型
- **工作**：点训练。平台先用预训练网络找出候选框，再把你框里的内容裁出来，教模型「认每个框是什么」。
- **原理**：**检测 = 找候选框（预训练帮你做）+ 给框分类（你来训练）**。这里还会自动加入「背景」类，让模型学会忽略无关的框。（对照：真正的 YOLO 是端到端一次学会「找+认」，但需要显卡——见深度学习地图。）
- **效果**：得到一个能给候选框贴标签的模型，几秒训完。

### 第 3 步 · 测试效果
- **工作**：上传一张新图，看模型画出的框对不对。
- **原理**：好检测要同时满足三点——**找得到**（不漏检）、**框得准**（位置贴合，用 IoU 衡量）、**认得对**（标签 + 置信度）。多个重叠框会用 **NMS** 去重。
- **效果**：直观看到漏检 / 误检 / 框偏，知道回第 1 步补哪类数据。

### 第 4 步 · 导出材料
- **工作**：一键打包可运行的检测脚本 + 模型 + 数据 + 说明。
- **原理**：导出的 `run.py` 完整复现「提框 → 裁剪 → 分类 → NMS → 画框」。
- **效果**：在自己电脑 / 行空板上跑通，验证模型真的能用。

## 5. 架构设计

### 5.1 新增 / 改动

```
app/ml/
  object_detector.py        已有：SSD/Haar 推理。新增「类别无关候选框」函数 propose_boxes()（复用 SSD）。
  detect_trainer.py  (新)   方案A 核心：裁剪框→MobileNet特征(复用 image_classifier/pretrained)→sklearn 分类器；
                            train()/predict()；负样本（背景类）生成；NMS。
  engine.py                 加 detection 能力分支（train/predict/可能的 compare）。
app/services/
  project_service.py        检测数据的存取（detect_images/、detect_labels.json）；训练/预测编排。
  dataset_library.py        加 COCO→平台框 的导入（解锁外部 6 个检测数据集）。
  template_service.py       渲染检测导出包（run.py：提框→裁剪→分类→NMS；bundle SSD onnx + 框分类器）。
app/models.py               TaskDefinition 已够用；新 ai_capability = "object_detector_trainable"（区别于演示）。
app/task_catalog.py         在 GENERAL_ML 加一个「目标检测」通用任务（+ 教育字段：生活例子/误区/实验/下一步/step_guides）。
templates/
  project.html              检测任务时，第1步挂标注工具、第3步挂画框结果（条件渲染或独立片段 project_detect 区块）。
static/
  detect.js  (新)           canvas 画框标注 + 画预测框；对齐 app.js 风格，做好后在 project 页按能力加载。
  styles.css                标注/画框相关样式。
app/api.py                  /api/projects/{id}/data/detect（存框）、train、predict/detect、export 分支。
```

### 5.2 与现有前端「对齐 + 整合」的做法

- 先把检测的前端（`detect.js` + 模板片段）做成**自洽的一块**，视觉、交互、文案都对齐现有四步（同样的 panel / 步骤锁 / 教育块 / 报告卡样式）。
- 完成后，在 `project.html` 里按 `task.ai_capability` 条件挂载：分类任务走现有 `app.js` 编辑器，检测任务走 `detect.js` 标注器，**共用四步骨架与教育框架**。
- 这样「独立开发、最后整合」两不误。

### 5.3 在线模型 ↔ 导出包一致性（平台核心不变量）

- 框分类器存 `{model, feature_mode, classes}`（含「背景」类）。
- 候选框、裁剪、MobileNet 特征预处理在 `detect_trainer.py` 与导出模板里**完全一致**（和图像分类同一套 `pretrained.py`）。
- 导出包 bundle：`models/<cap>.joblib` + `models/pretrained/ssd.onnx` + `mobilenet_v2.onnx`。
- 新增 `tests/`：执行导出的检测 `run.py`，对一张图跑出框（沿用 `test_generation_services.py` 的子进程模式）。

- **不新增重依赖**（方案 A）：SSD onnx + onnxruntime + Pillow + numpy + sklearn 都已在用。
- `models_pretrained/`（已有，gitignored）必须有 SSD + MobileNet onnx；缺失时检测模块友好降级 + 安装提示（和现有一致）。
- 方案 B（YOLO）若做，隔离进 `.[detect]` extra（Ultralytics / torch），**绝不进 base**。

## 7. 工作量分解（建议分期）

| 期 | 内容 | 依赖 |
|---|---|---|
| **一期（MVP）** | `propose_boxes()` + `detect_trainer.py`（裁剪+特征+分类+背景类+NMS）；后端 train/predict；最简标注工具（画框/删框/贴标签）；第3步画预测框；一个通用检测任务 + 四步教育内容 | SSD/MobileNet onnx 已有 |
| **二期** | COCO 导入器（解锁 6 个数据集）；导出包（run.py 检测）+ 子进程测试；标注工具打磨（移动/缩放/快捷键） | 一期 |
| **三期（可选）** | 第0步「先体验」入口（方案C，升级 playground）；候选框兜底（滑窗/selective-search）；导出包里的 YOLO 进阶脚本（方案B 彩蛋） | 二期 |
| **教育增强** | 检测专题接入「深度学习地图」（检测 vs 分类、IoU、NMS、YOLO 为何要显卡）；标注质量/漏检误检的实验卡片 | 一期 |

每一期都按现有节奏：小步、可测、子 agent 审中文写作、浏览器验证、CHANGELOG + 版本号、commit/push。

## 8. 风险与开放问题

| 问题 | 说明 / 建议 |
|---|---|
| 候选框召回 | SSD 对非 COCO 目标可能提不出框 → 一期先承认局限并作为教学点；二期加兜底。 |
| 背景负样本怎么取 | 取与所有标注框 IoU 很低的随机区域当「背景」；数量与正样本平衡。需调参。 |
| 标注工具工作量 | canvas 画框是本模块最大前端件，建议一期只做「画/删/贴标签」，移动缩放放二期。 |
| 课堂时间 | 标注比贴标签慢得多。建议每个任务配好「导入现成检测数据集」入口，降低标注门槛。 |
| 评估指标 | 一期用「框对/漏/误」的直观呈现即可，IoU/mAP 作为进阶讲解，不强求数字指标。 |
| 能力命名 | 新 `ai_capability` 名待定（如 `object_detector_trainable`），与演示用的检测区分开。 |
| 与竞赛的关系 | 本模块先做通用教育版；竞赛检测题日后在独立竞赛页整合（与平台整体方向一致）。 |

## 9. 成功指标（定性）

1. 学生能说出**检测和分类的区别**：「分类说是什么，检测还说在哪」。
2. 学生能复述**平台的检测原理**：「先找候选框，再认每个框是什么」。
3. 学生标注几张图、训练几秒，就能在新图上看到**自己模型画的框**。
4. 学生能指出一次结果的**漏检 / 误检 / 框偏**，并知道回第 1 步补哪类数据。
5. 学生理解**为什么真检测（YOLO）要显卡**，而平台用了一条 CPU 也能走的取巧路径。

---

## 10. 一句话总结

**用「预训练找框 + 学生训练的框分类器」这条 CPU 友好的取巧路径，把目标检测装进现有四步骨架，每步配教育内容；先做通用教育版 MVP，真 YOLO 作为隔离的进阶彩蛋。** 既守住课堂体验与平台不变量，又把检测的原理讲透。
