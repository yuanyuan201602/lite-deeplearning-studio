# 目标检测模块 PRD / 技术方案 — Lite DeepLearning Studio

> 版本：v1 · 2026-06-13（2026-06-14 更新进度）
> 状态：**一期已实现**（骨架 + 标注工具 + 算法 A「轻量检测」，v0.6.0）。**二期·导出包已实现**（算法 A 的可运行检测材料包，v0.7.0）。二期剩余：算法 B（YOLO）。三期（COCO 导入）待做。

---

## 0. 定位与约束

用户决定：**把目标检测当作一个独立模块来做**，前端对齐现有四步风格，做好后整合进现有前端；过程中**每一步都嵌入教育内容**，让学生理解四步里每一步的「工作 / 原理 / 效果」。

这份方案要先回答一个绕不开的硬约束，再谈怎么做。

### 硬约束：CPU 课堂里怎么「训练」检测？

平台的灵魂是 **CPU 友好、课堂里几秒到几十秒出结果、不需要显卡**（见 `agent.md`：「对象检测，除非某个竞赛任务明确需要，否则推迟完整 YOLO 集成」）。

但「真正的目标检测训练」（YOLO / SSD 微调）：
- 需要 GPU；在 CPU 上微调一个检测模型要**几十分钟到几小时**；
- 依赖重（PyTorch / Ultralytics，~2GB）；
- 一次训练几十个 epoch，完全没有「点一下、几秒出结果」的课堂感。

**直接照搬「真检测训练」会破坏平台体验。** 但也不能因此就不给学生「真 YOLO」。

所以——**像分类模块给多个算法选择那样，检测模块也提供两种「都真实可运行」的算法，让学生按场景选、并讲清各自优劣**（不做「全部算法一起跑」的对比，只把两者区别讲清楚）：

- **轻量检测**（候选框 + 框分类器，下文算法 A）：CPU 友好、几秒可训，是「点一下出结果」的课堂主力；本质是经典 R-CNN 的简化版。
- **YOLO 端到端检测**（下文算法 B）：工业界主流的真检测，端到端同时学「找框 + 认框」；更强，但更重、训练更慢、建议显卡。

两者**都是真实可运行**的模块（不是一真一假），共用同一套四步骨架、标注工具与数据。

## 1. 检测和分类的根本区别（为什么不能直接套用四步）

| | 图像分类（已有） | 目标检测（本模块） |
|---|---|---|
| 回答的问题 | 「这**是**什么？」 | 「**有哪些**东西、各自**在哪**？」 |
| 输出 | 一个类别 | 多个框（位置 x/y/w/h）+ 每个框的类别 + 置信度 |
| 标注 | 给整张图贴一个标签 | 在图上**画框**并给每个框贴标签 |
| 评估 | 准确率 | 框准不准（IoU）、漏检 / 误检、置信度 |
| 难点 | 认得准 | 既要找得到、又要框得准、还要认得对 |

→ 四步流程的**骨架可以复用**（准备数据 → 训练 → 测试 → 导出），但**每一步的内部都要重做**：第 1 步要画框标注工具，第 2 步是不同的训练逻辑，第 3 步要在图上画预测框，第 4 步导出检测脚本。这正是「独立模块」的原因。

## 2. 两个算法：都真实可运行，讲清区别而非比拼

检测模块提供**两种算法作为「训练」步骤的选择**（沿用分类模块「算法卡片」的形式，但**只讲区别、不跑「全部一起跑」的对比**——两者特性差太远，赛跑没意义）。两者共用同一套标注数据，都能真实训练、运行、导出。

### 算法 A · 轻量检测（候选框 + 框分类器）— 经典 R-CNN 简化版

把检测拆成两步，**复用平台已有的图像分类基建**：

```
找候选框（借用预训练 SSD，不训练）  →  认每个框是什么（学生训练，CPU 几秒）
```

流程：SSD 提候选框 → 学生标注 → 裁剪每个框 → MobileNet 提特征（复用 `image_classifier` 那套）→ sklearn 分类器（外加「背景」类拒绝无关框）→ 预测时 提框→裁剪→分类→NMS→画框 → 导出复用 `ai_runtime/core.py` 模式。

- **本质**：这就是 **R-CNN（2014）** 的简化版——检测最早的主流做法（候选框 + 裁剪 + CNN 特征 + 分类器 + NMS）。**真检测，经典流派，不是模拟。**
- **优**：CPU 几秒可训、零新增重依赖、原理好懂、守住「在线↔导出」一致性。
- **劣**：「找框」是借来的（学生没训练定位）、只能检测 SSD 提得出框的目标、非端到端、精度有限。
- **依赖**：base（SSD + MobileNet onnx，已有）。

### 算法 B · YOLO 端到端检测 — 工业级真检测

用 Ultralytics YOLOv8n 在学生标注的数据上**真微调**：一个网络**同时学会「找框 + 认框」**，端到端。

- **本质**：现代主流检测器。**找框也是学出来的**，不借用别人。真正端到端。
- **优**：精度高、能检测自定义目标、工业界在用、学生真正训练出一个完整检测器。
- **劣**：依赖重（`.[detect]`：Ultralytics/torch，~2GB）、CPU 上训练慢（分钟级起，建议显卡）、训练 UX 与「几秒出结果」不同（需进度条 / 轮次）。
- **依赖**：`.[detect]` 可选组件；未安装时算法卡片可见但选中给安装提示（与 `.[ocr]` 同款降级），**绝不进 base**——学生不装也能用算法 A 走完整套四步。

### 为什么是「选择 + 讲区别」，不做「比拼」

A 和 B 不是同一量级的竞争者，而是**检测技术演进的两端**：A = 早期 R-CNN 思路（拆成两步、借候选框），B = 现代 YOLO（端到端、自己学找框）。让学生**两个都跑一遍、对照体会**，比一个准确率数字有用得多——这正好把「深度学习地图」里的技术演进做成了可动手的对照。所以检测**不做** classifier 那种「对比所有模型」赛跑，只在算法卡片里把两者的工作原理与优劣讲清楚。

### 第 0 步「先体验」（可选保留）

现有 `/playground/detect`（预训练 SSD 直接识别 COCO 80 类）可作为进入检测任务前「先看看检测长什么样」的入口，不属于四步主体。

## 3. 数据与标注

### 3.1 数据格式（对齐已整理的检测数据集）

外部目录有 6 个检测数据集，均为 **COCO 格式**（已确认）：
```
<dataset>/
  classes.txt                  # 每行一个类别名
  images/train/*.jpg  images/valid/*.jpg
  annotations/train.json       # COCO: images[] + annotations[bbox,category_id,image_id] + categories[]
  annotations/valid.json
```
- `bbox = [x, y, w, h]`（像素坐标）。
- 平台可写一个 COCO → 平台框存储 的导入器，**复用 `dataset_library` 那套「从整理好的数据集导入」**，把这 6 个数据集接进来（数据侧的「暂不支持」就此解锁）。

### 3.2 平台内框存储（新）

```
workspace/projects/<id>/dataset/
  detect_images/<uuid>.jpg            # 上传的原图
  detect_labels.json                  # [{image, boxes:[{x,y,w,h,label}], width, height}]
```

### 3.3 标注工具（本模块最大的新前端件）

- 在第 1 步：上传图片 → 在 `<canvas>` 上**拖拽画框** → 给框选类别（复用类别管理 UI）→ 保存。
- 支持：画框、移动 / 删除框、缩放适配、每类不同颜色、键盘删除。
- 对齐现有 `static/app.js` / `collect.js` 的风格，独立成 `static/detect.js`，做好后整合。

## 4. 四步流程 × 教育内容

每一步都配「了解这一步」（工作 / 原理 / 效果），和现有任务的 `step_guides` 一致风格。

### 第 1 步 · 准备数据（画框标注）
- **工作**：上传图片，在每张图上把目标**框出来**并贴标签；或一键导入整理好的检测数据集。
- **原理**：检测比分类多了「在哪」。框（bounding box）就是「在哪」的答案；框得准不准、贴得全不全，直接决定模型上限。漏标一个目标，模型就学不会它。
- **效果**：得到一批「图 + 框 + 标签」，这是检测训练的原料。

### 第 2 步 · 训练模型
- **工作**：先选算法（轻量检测 / YOLO，算法卡片讲清区别），再点训练。
- **原理**：
  - 选**轻量检测**：平台先用预训练网络找候选框，再把你框里的内容裁出来、教模型「认每个框是什么」（外加「背景」类忽略无关框）——这是 R-CNN 思路，CPU 几秒。
  - 选 **YOLO**：一个网络端到端**同时学会「找框 + 认框」**，更强但更慢、需 `.[detect]` 组件、建议显卡。
  - 两个都跑一遍对照，正好理解检测从 R-CNN 到 YOLO 的演进（见深度学习地图）。
- **效果**：得到一个真能在新图上画框的检测模型。

### 第 3 步 · 测试效果
- **工作**：上传一张新图，看模型画出的框对不对。
- **原理**：好检测要同时满足三点——**找得到**（不漏检）、**框得准**（位置贴合，用 IoU 衡量）、**认得对**（标签 + 置信度）。多个重叠框会用 **NMS** 去重。
- **效果**：直观看到漏检 / 误检 / 框偏，知道回第 1 步补哪类数据。

### 第 4 步 · 导出材料
- **工作**：一键打包可运行的检测脚本 + 模型 + 数据 + 说明。
- **原理**：导出的 `run.py` 完整复现「提框 → 裁剪 → 分类 → NMS → 画框」。
- **效果**：在自己电脑 / 行空板上跑通，验证模型真的能用。

## 5. 架构设计

### 5.1 新增 / 改动

```
app/ml/
  object_detector.py        已有：SSD/Haar 推理。新增「类别无关候选框」函数 propose_boxes()（复用 SSD）。
  detect_lite.py     (新)   算法A：裁剪框→MobileNet特征(复用 image_classifier/pretrained)→sklearn 分类器；
                            train()/predict()；负样本（背景类）生成；NMS。base 依赖。
  detect_yolo.py     (新)   算法B：Ultralytics YOLOv8n 微调 train()/predict()；隔离在 .[detect]，未装给安装提示。
  engine.py                 检测能力按所选算法(lite/yolo)分派 train/predict；检测不做 compare。
app/services/
  project_service.py        检测数据的存取（detect_images/、detect_labels.json）；训练/预测编排。
  dataset_library.py        加 COCO→平台框 的导入（解锁外部 6 个检测数据集）。
  template_service.py       渲染检测导出包（run.py：提框→裁剪→分类→NMS；bundle SSD onnx + 框分类器）。

app/task_catalog.py         在 GENERAL_ML 加一个「目标检测」通用任务（+ 教育字段：生活例子/误区/实验/下一步/step_guides）。
templates/
  project.html              检测任务时，第1步挂标注工具、第3步挂画框结果（条件渲染或独立片段 project_detect 区块）。
static/
  detect.js  (新)           canvas 画框标注 + 画预测框；对齐 app.js 风格，做好后在 project 页按能力加载。
  styles.css                标注/画框相关样式。
app/api.py                  /api/projects/{id}/data/detect（存框）、train、predict/detect、export 分支。
```

### 5.2 与现有前端「对齐 + 整合」的做法

- 先把检测的前端（`detect.js` + 模板片段）做成**自洽的一块**，视觉、交互、文案都对齐现有四步（同样的 panel / 步骤锁 / 教育块 / 报告卡样式）。
- 完成后，在 `project.html` 里按 `task.ai_capability` 条件挂载：分类任务走现有 `app.js` 编辑器，检测任务走 `detect.js` 标注器，**共用四步骨架与教育框架**。
- 这样「独立开发、最后整合」两不误。

### 5.3 在线模型 ↔ 导出包一致性（平台核心不变量）

- **算法 A**：框分类器存 `{model, feature_mode, classes}`（含「背景」类）；候选框、裁剪、MobileNet 特征预处理在 `detect_lite.py` 与导出模板里**完全一致**（和图像分类同一套 `pretrained.py`）；导出 bundle `joblib + ssd.onnx + mobilenet_v2.onnx`。
- **算法 B**：导出训练好的 YOLO 权重（.pt 或导出的 onnx）+ run.py（用 ultralytics 或 onnxruntime 推理），导出包注明所需组件。
- 新增 `tests/`：执行导出的检测 `run.py`，对一张图跑出框（沿用 `test_generation_services.py` 的子进程模式）；算法 B 的子进程测试需 `.[detect]`，CI 上可标 skip。

## 6. 依赖

- **算法 A 不新增重依赖**：SSD onnx + onnxruntime + Pillow + numpy + sklearn 都已在用。
- `models_pretrained/`（已有，gitignored）必须有 SSD + MobileNet onnx；缺失时算法 A 友好降级 + 安装提示（和现有一致）。
- **算法 B（YOLO）作为第二个真实算法**：隔离进 `.[detect]` extra（Ultralytics / torch，~2GB），未安装时算法卡片提示安装（同 `.[ocr]` 降级），**绝不进 base**。学生不装也能用算法 A 走完整套四步。

## 8. 风险与开放问题

| 问题 | 说明 / 建议 |
|---|---|
| 候选框召回 | SSD 对非 COCO 目标可能提不出框 → 一期先承认局限并作为教学点；二期加兜底。 |
| 背景负样本怎么取 | 取与所有标注框 IoU 很低的随机区域当「背景」；数量与正样本平衡。需调参。 |
| 标注工具工作量 | canvas 画框是本模块最大前端件，建议一期只做「画/删/贴标签」，移动缩放放二期。 |
| 课堂时间 | 标注比贴标签慢得多。建议每个任务配好「导入现成检测数据集」入口，降低标注门槛。 |
| 评估指标 | 一期用「框对/漏/误」的直观呈现即可，IoU/mAP 作为进阶讲解，不强求数字指标。 |
| 能力命名 | 一个检测能力（如 `object_detector_trainable`）+ `model_choice` 选 lite / yolo，而不是两个 capability。 |
| YOLO 训练时长 / UX | CPU 上分钟级，需进度条 / 轮次提示，和「几秒 + 终端日志」不同；二期单独设计训练 UX，默认小模型(yolov8n) + 少轮次。 |
| 与竞赛的关系 | 本模块先做通用教育版；竞赛检测题日后在独立竞赛页整合（与平台整体方向一致）。 |

## 9. 成功指标（定性）

1. 学生能说出**检测和分类的区别**：「分类说是什么，检测还说在哪」。
2. 学生能复述**平台的检测原理**：「先找候选框，再认每个框是什么」。
3. 学生标注几张图、训练几秒，就能在新图上看到**自己模型画的框**。
4. 学生能指出一次结果的**漏检 / 误检 / 框偏**，并知道回第 1 步补哪类数据。
5. 学生**两个算法都跑过**，能说出轻量检测（R-CNN 思路、CPU 快）和 YOLO（端到端、更强但要算力）各自的优劣与适用场景。

## 10. 一句话总结

**检测模块提供两种「都真实可运行」的算法作为训练选择——轻量检测（R-CNN 简化版，CPU 几秒）和 YOLO（端到端，重 / 慢 / 建议显卡），讲清区别而非比拼；共用同一套四步骨架、标注工具与教育内容。先做骨架 + 算法 A（一期），再接入 YOLO（二期）。** 既守住课堂体验（A 走主路径）、又给到工业级真检测（B），还把检测从 R-CNN 到 YOLO 的演进做成了可动手的对照。
