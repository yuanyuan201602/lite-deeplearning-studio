## 2. 产品定位

Lite DeepLearning Studio 是一个面向中小学生 AI 竞赛的本地优先轻量工作流平台。

它不做大型云端训练平台，也不做自动替学生参赛的智能体系统。最终版要解决的是一个更具体的问题：

> 学生和老师可以根据比赛要求，快速选择任务、替换真实数据、完成本地训练与推理、接入基础语音/硬件模板，并导出可提交、可调试、可验收的项目任务包。

轻量化工作流版本的核心价值是“把比赛任务拆成可执行步骤”，而不是堆叠复杂功能。

## 3. 当前进度基线

当前项目已经具备以下能力：

- 本地 FastAPI + Jinja2 Web 应用。
- 首页展示两个比赛入口：智能博物、优创未来。
- 任务目录覆盖两份比赛文件中的主要任务。
- 每个任务可生成 zip 导出包。
- 导出包包含训练脚本、预测脚本、运行脚本、Notebook、样例数据、AI runtime、硬件说明、提交说明和竞赛核对清单。
- 文本分类、句子分类、图像分类、知识问答检索、传感器决策可在本地样例数据上训练和推理。
- OCR/错字检查使用 EasyOCR，环境不可用时明确报告失败，不伪装通过。
- 智能博物任务包生成 DFRobot 语音合成模块播报模板。
- 优创未来小学/初中任务包生成行空板关键词语音互动模板。
- 优创未来高中任务包生成 Jetson 外接麦克风语音智能体接口模板。
- 已有 30 个自动化测试和验收脚本，覆盖任务目录、导出包结构、AI 样例运行和 Web 生成流程。
- 已有 GitHub 仓库化说明和学生机安装说明；远程仓库发布、release、安装包制作和干净机器验证仍属于最终交付动作。

当前仍未完成：

- 学生真实数据上传、替换、管理和二次训练流程。
- Web 端逐步引导式调试体验。
- 摄像头拍照、麦克风录音、串口播报等真实硬件闭环。
- 行空板真实语音识别联调。
- Jetson 高中组真实 ASR/TTS/大模型智能体联调。
- 机械装置控制闭环。
- 面向提交材料的一键整理和检查体验。
- GitHub 远程仓库发布和版本 release。
- 学生机安装包制作、分发和实机安装验收。

## 4. 最终版目标

轻量化工作流最终版应达到以下状态：

1. 学生能从比赛任务出发，而不是从空白代码出发。
2. 每个任务都有明确的“准备数据、训练、测试、硬件迁移、提交材料”步骤。
3. 学生可以在 Web 页面中替换样例数据为真实数据。
4. 非 OCR AI 任务可以稳定完成本地训练和推理。
5. OCR 任务在安装 EasyOCR/torch/opencv 的增强环境中可以完成图片文字识别和错字定位；基础环境必须明确报告依赖缺失。
6. 语音互动按比赛需要提供可落地的轻量方案：
   - 智能博物：DFRobot 语音合成模块播报。
   - 优创未来小学/初中：行空板内置麦克风关键词识别。
   - 优创未来高中：Jetson Super Nano 外接麦克风接口和大模型智能体扩展位。
7. 每个导出包都能被老师按清单检查是否满足比赛要求。
8. 平台可以连续多轮生成和验收任务包，适合作为训练营或赛前辅导工具。
9. 项目可以发布为 GitHub 仓库，便于版本管理、协作开发和后续交付。
10. 最终版本可以打包为学生机可安装版本，降低课堂和赛前部署成本。

## 5. 非目标

轻量化工作流最终版不包含以下内容：

- 多用户账号、班级管理、云端教师后台。
- 云端模型训练、GPU 资源调度、实验追踪平台。
- 自动生成完整参赛作品并替代学生设计。
- 通用自主编程 Agent。
- YOLO、Whisper、本地 LLM 的完整生产级集成。
- 机械臂、舵机、传送带等具体硬件的通用控制平台。
- 真实比赛现场自动适配所有临时题目。

这些能力可以作为后续专业版或硬件版路线，不纳入轻量版验收。

说明：

- GitHub 仓库发布属于轻量版交付范围，但不包含复杂 CI/CD、多人权限治理或正式开源社区运营。
- 学生机安装版本属于轻量版交付范围，但优先支持 macOS/Windows 普通安装和本地启动，不承诺覆盖所有 Linux 发行版和所有校园机房管控环境。

## 6. 用户与场景

### 6.1 学生

学生需要在较短时间内把比赛要求转成能运行的项目。

典型场景：

- 选择“智能博物挑战三”，录入非遗类别，替换训练文本，训练分类模型，检查输出格式是否为“XXX属于XXX”。
- 选择“优创未来初中组”，录入现场关键词，上传两类卡片图片，训练视觉模型，生成行空板语音模板和提交材料清单。
- 选择“优创未来高中组”，先完成视觉识别和语音接口，再把大模型和机械装置作为后续硬件扩展接入。

### 6.2 指导教师

教师需要快速判断学生作品是否覆盖比赛要求。

典型场景：

- 检查每个学生导出的 `docs/competition_checklist.md`。
- 运行 `python run.py` 查看 AI 能力是否真实运行。
- 查看 `outputs/ai_validation_result.json` 判断训练、推理和 OCR 环境状态。
- 查看 `submission/README.md` 确认创作说明、视频、照片、代码是否齐全。

### 6.3 竞赛辅导者

辅导者需要批量组织训练任务。

典型场景：

- 为小学、初中、高中分别生成标准项目包。
- 使用样例数据演示完整流程。
- 要求学生替换真实数据后再次运行。
- 用验收脚本连续检查多个任务包。

## 8. 核心工作流

### 8.1 选择任务

用户打开首页后，应先看到比赛和任务，而不是营销介绍。

要求：

- 首页展示“智能博物”和“优创未来”。
- 每个比赛展示任务卡片。
- 每张任务卡片显示年龄组、任务目标、AI 能力状态。
- 已暂停或需硬件联调的能力必须显式标记。

### 8.2 配置任务

用户进入任务页后填写少量必要信息。

必填：

- 项目名称。
- 目标任务。
- 目标硬件。

可选：

- 学生姓名。
- 类别、关键词或问答主题。
- 数据说明。
- 真实数据上传。
- 语音播报文本。
- 硬件端口或设备说明。

要求：

- 页面要显示当前任务的比赛要求。
- 页面要显示将生成哪些文件。
- 页面要显示真实 AI 能力状态：可训练、可推理、需 OCR 依赖、本轮只提供接口。

### 8.3 数据准备

最终版必须从“只有样例数据”升级为“样例数据 + 真实数据替换”。

要求：

- 文本任务支持上传 CSV 或直接粘贴文本。
- 图像任务支持按类别上传图片。
- OCR 任务支持上传知识卡片图片和标准答案文本。
- QA 任务支持上传问答表或手动录入问答对。
- 传感器任务支持上传 CSV 或填写输入输出样例。
- 如果用户未上传真实数据，平台继续生成样例数据，但必须在结果中标记“当前使用样例数据”。

### 8.4 训练与推理

导出包内必须继续保持离线可运行。

要求：

- `python run.py` 能完成训练、预测和验证报告生成。
- `train.py` 只负责训练。
- `predict.py` 只负责推理。
- `ai_runtime/core.py` 放通用能力实现。
- `outputs/ai_validation_result.json` 记录运行结果。
- `docs/ai_validation.md` 解释如何判断 AI 能力是否真实运行。

### 8.5 语音与硬件迁移

语音能力采用轻量、分层落地策略。

智能博物：

- 默认使用 DFRobot 语音合成模块完成播报。
- 任务包生成串口 TTS 示例代码。
- 支持无串口环境下打印播报文本，便于电脑端测试。

优创未来小学/初中：

- 默认使用行空板内置麦克风识别关键词。
- 任务包生成关键词配置和识别接口。
- 允许用户现场替换关键词。

优创未来高中：

- 默认面向 Jetson Super Nano + 外接麦克风。
- 最终轻量版只要求生成语音智能体接口、配置说明和测试占位。
- 真实大模型、ASR/TTS、机械控制不作为轻量版强制通过项。

硬件说明：

- 每个任务包必须包含 `hardware/README.md`。
- 必须说明电脑端验证、目标硬件迁移、端口配置、常见问题。
- 暂停项必须写入说明和核对清单。

### 8.6 导出与提交

生成结果页必须让用户清楚知道自己得到了什么。

导出包必须包含：

- `README.md`
- `train.py`
- `predict.py`
- `run.py`
- `notebook.ipynb`
- `ai_runtime/`
- `data_sample/`
- `models/`
- `outputs/`
- `requirements.txt`
- `hardware/README.md`
- `speech/README.md`
- `speech/speech_output.py`
- `speech/voice_config.json`
- `submission/README.md`
- `docs/competition_checklist.md`
- `docs/ai_validation.md`

提交材料说明必须覆盖：

- 创作说明。
- 程序代码。
- 作品照片。
- 演示视频。
- 现场展示核对项。
- AI 能力验证结果。
- 暂停或需人工完成的硬件项。

## 9. 功能需求

### 9.1 任务目录

优先级：P0

需求：

- 任务定义必须覆盖当前两份比赛文件中的主要任务。
- 每个任务必须声明：
  - `ai_capability`
  - `sample_dataset_kind`
  - `runtime_requirements`
  - `voice_profile`
  - `paused_features`
  - `competition_requirements`
  - `required_outputs`
- 任务目录变更必须有测试覆盖。

验收：

- 所有任务页面可打开。
- 所有任务可生成导出包。
- 测试能发现缺少 AI 能力声明的任务。

### 9.2 模板生成

优先级：P0

需求：

- 根据 `ai_capability` 选择不同模板。
- 生成任务相关样例数据。
- 生成训练、推理、验证、硬件和提交文档。
- 保持导出包结构稳定。

验收：

- 每类 AI 能力至少有一个任务包可运行。
- 非 OCR 任务 `python run.py` 必须成功。
- OCR 任务在 EasyOCR 可用时必须成功，不可用时必须明确失败；`--require-ai` 不允许 OCR 任务用降级文本冒充通过。

### 9.3 数据替换

优先级：P0

需求：

- Web 端支持真实数据上传或录入。
- 导出包保留样例数据，但优先使用用户数据。
- 用户数据格式错误时给出清楚提示。
- 生成结果中标明“样例数据”或“真实数据”。

验收：

- 文本、图像、OCR、QA、传感器五类数据均能完成一次真实数据导入。
- 导入后的任务包仍可执行 `python run.py`。

### 9.4 AI 本地运行

优先级：P0

需求：

- 核心 AI 依赖保持 CPU 可运行。
- 文本分类使用 TF-IDF + scikit-learn 分类器。
- 图像分类使用 Pillow/numpy 特征 + scikit-learn 分类器。
- QA 使用 TF-IDF 相似度检索。
- 传感器决策使用决策树/规则混合。
- OCR 使用 EasyOCR，作为重型可选能力。

验收：

- `scripts/acceptance_check.py --rounds 5 --require-ai` 通过。
- 验收报告包含每个任务包的训练、推理和输出文件检查结果。

### 9.5 语音模板

优先级：P1

需求：

- 智能博物生成 DFRobot TTS 播报模板。
- 优创未来小学/初中生成行空板关键词识别模板。
- 优创未来高中生成 Jetson 语音智能体接口模板。
- 语音模板必须有电脑端可测试 fallback。
- 语音配置必须写入 `speech/voice_config.json`。

验收：

- 每个任务包都包含 `speech/README.md`、`speech/speech_output.py`、`speech/voice_config.json`。
- DFRobot、行空板、Jetson 三类模板均有自动化测试覆盖。

### 9.6 Web 体验

优先级：P1

需求：

- 首页、任务页、结果页保持简洁。
- 任务页要更像“工作流”，而不是代码生成表单。
- 结果页显示 AI 能力状态、数据来源、导出文件、下一步。
- 错误页要用学生能理解的语言说明原因。

验收：

- 老师可以在 3 分钟内完成一次任务包生成。
- 学生可以按页面提示知道下一步要替换什么数据。
- 生成失败时能定位是数据问题、依赖问题还是系统问题。

### 9.7 验收与报告

优先级：P0

需求：

- 保留并增强 `scripts/acceptance_check.py`。
- 支持多轮验收。
- 支持检查导出包文件、AI 运行、OCR 状态、语音模板、提交清单。
- 支持 `--with-web` 检查首页、任务页、表单生成和 zip 下载。
- 生成机器可读 JSON 报告。

验收：

- `pytest -q` 通过。
- `ruff check app tests scripts` 通过。
- `scripts/acceptance_check.py --rounds 5 --require-ai` 通过。
- `scripts/acceptance_check.py --rounds 5 --require-ai --with-web` 在 OCR 增强环境中通过。
- 基础环境可运行 `scripts/acceptance_check.py --rounds 1 --with-web`，并应在缺少 OCR 增强依赖时让 `--require-ai` 明确失败。

### 9.8 GitHub 仓库化

优先级：P1

需求：

- 将项目整理为可推送到 GitHub 的标准仓库。
- 仓库根目录必须包含清晰的 `README.md`、`AGENTS.md`、`pyproject.toml` 和 `docs/`。
- 补充 `.gitignore`，排除缓存、虚拟环境、生成的 workspace 项目包、临时验收输出和大型本地模型文件。
- 保留必要的示例文档和测试代码。
- 不提交本地隐私数据、学生真实数据、PDF 原件中不应公开的内容、环境密钥或账号信息。
- 用户提供 GitHub 账号和仓库归属后，再执行远程仓库创建、首次提交和推送。

验收：

- 本地仓库初始化成功。
- `git status` 只包含预期项目文件。
- 首次提交采用 Conventional Commits 格式。
- GitHub 远程仓库可访问。
- 新环境可以从 GitHub 拉取代码并按 README 启动。

### 9.9 学生机可安装版本

优先级：P0

需求：

- 最终版必须提供学生机安装方式，便于课堂、训练营和赛前集中部署。
- 安装包优先目标是“少配置、能启动、能生成任务包”。
- 支持至少一种稳定安装形态：
  - Windows/macOS 一键安装包或启动器。
  - 或基于 Python 运行时的离线安装包。
  - 或带依赖锁定的压缩包分发方案。
- 安装后学生可以通过桌面入口或本地启动脚本打开平台。
- 安装包必须包含核心 AI 依赖。
- OCR 依赖可作为增强安装项，安装失败时不影响非 OCR 任务使用。
- 安装包必须包含使用说明、卸载说明和常见问题。
- 学生机安装版本默认本地运行，不依赖公网服务。

验收：

- 在一台干净学生机上完成安装。
- 安装后可打开本地 Web 页面。
- 至少能生成一个智能博物任务包和一个优创未来任务包。
- 非 OCR AI 任务可运行 `python run.py` 或等价启动命令。
- OCR 增强项安装成功时 OCR 任务可运行；未安装时平台明确提示。
- 卸载或删除目录后不残留关键运行依赖和临时项目文件。

### 12.2 AI 依赖

- 老师或学生电脑本地启动 Web 应用。
- 生成任务包后迁移到行空板、Jetson、树莓派或其他设备。
- 不要求公网访问。

### 12.4 仓库与版本管理

- 使用 Git 管理项目版本。
- GitHub 仓库作为代码备份、版本发布和后续协作入口。
- 默认分支建议使用 `main`。
- 标签建议使用 `v0.1.0`、`v0.2.0`、`v1.0.0` 这类语义化版本。
- 重要交付版本必须有 release notes，说明新增能力、安装方式、已知问题和测试结果。

### 12.5 安装包形态

学生机安装版本建议分层：

- 基础版：Web 应用 + 核心 AI 依赖，支持文本、图像、QA、传感器任务。
- OCR 增强版：基础版 + EasyOCR/torch/opencv-python-headless。
- 开发版：包含测试、验收脚本和开发依赖，供老师或维护者使用。

安装包必须尽量避免学生手动处理复杂依赖。若无法完全避免，安装说明必须提供明确步骤和常见错误处理。

## 13. 验收标准

### 13.1 产品验收

- 首页展示两个比赛和所有任务。
- 每个任务能进入工作流页。
- 每个任务能生成 zip 导出包。
- 导出包文件结构完整。
- 每个导出包包含比赛核对清单。
- 每个导出包包含提交材料说明。
- 用户能分辨当前任务使用的是样例数据还是真实数据。

### 13.2 AI 验收

- 文本分类任务能训练并预测。
- 句子分类任务能训练并预测。
- 图像分类任务能训练并预测。
- 人脸/卡片识别任务能训练并预测。
- QA 检索任务能返回最相近答案。
- 传感器决策任务能输出动作。
- OCR 任务在 EasyOCR 可用时能识别图片并检查错字。
- OCR 任务在 EasyOCR 不可用时必须明确报告环境缺失。

### 13.3 语音验收

- 智能博物任务包包含 DFRobot TTS 方案。
- 优创未来小学/初中任务包包含行空板关键词语音方案。
- 优创未来高中任务包包含 Jetson 外接麦克风接口说明。
- 语音模板能在无硬件环境下进行文本 fallback 测试。

### 13.4 自动化验收

必须通过：

```bash
python -m pytest -q
python -m ruff check app tests scripts
python scripts/acceptance_check.py --rounds 5 --with-web
```

OCR 增强环境必须额外通过：

```bash
python scripts/acceptance_check.py --rounds 5 --require-ai --with-web
```

### 13.5 GitHub 仓库验收

- 仓库可以被克隆。
- 按 README 安装依赖后可以启动应用。
- 测试命令可以运行。
- 仓库不包含本地缓存、临时 workspace、大型生成包、账号密钥或学生隐私数据。
- release 页面至少包含一个可追踪版本说明。

### 13.6 安装版验收

- 在目标学生机上完成一次全新安装。
- 安装后能通过本地地址打开平台。
- 安装后能生成并下载任务包。
- 基础 AI 任务能完成本地训练/推理。
- OCR 增强能力有明确安装状态。
- 安装说明可以被非开发背景老师照着执行。

## 15. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| OCR/EasyOCR 安装较重 | 学生电脑可能失败 | 作为可选重依赖，失败时明确报告环境缺失 |
| 真实硬件差异大 | 串口、麦克风、驱动调试不可控 | 轻量版只承诺模板和最小联调，不承诺所有设备即插即用 |
| 高中组大模型要求复杂 | 超出轻量版范围 | 保留 Jetson 接口和任务说明，后续单独排期 |
| 现场题目临时发布 | 预置数据无法完全匹配 | 提供快速替换关键词、类别和数据的流程 |
| 学生误以为样例结果可直接参赛 | 作品质量风险 | 在结果页和文档中标记样例数据状态 |
| 任务包结构变动导致验收失效 | 维护成本上升 | 固定标准导出结构并用测试覆盖 |
| GitHub 仓库误提交隐私或大型文件 | 泄露风险和仓库膨胀 | 推送前检查 `.gitignore`、文件清单和历史记录 |
| 学生机环境差异大 | 安装失败或启动失败 | 提供基础版、OCR 增强版和离线包三层方案 |

## 16. 成功指标

轻量化工作流最终版完成后，应满足：

- 100% 任务可生成导出包。
- 100% 导出包包含标准文件结构。
- 100% 非暂停 AI 能力有本地运行验证。
- 100% 任务包含竞赛核对清单和提交材料说明。
- 老师可在 3 分钟内生成一个标准任务包。
- 学生可在 15 分钟内完成一次样例训练/推理。
- 使用真实数据替换后，核心任务可在本地完成训练和预测。
- 自动化验收连续 5 轮通过；OCR 增强环境额外通过 `--require-ai --with-web`。
- GitHub 仓库可克隆、可启动、可追踪版本。
- 学生机安装版可在干净机器上完成安装和基础任务生成。

## 17. 最终版一句话验收

当一个学生选择任意一个已覆盖比赛任务，能按页面步骤替换数据、生成任务包、在本地运行 `python run.py` 得到 AI 验证结果，并根据核对清单准备比赛提交材料时，轻量化工作流版本即达到最终效果。
